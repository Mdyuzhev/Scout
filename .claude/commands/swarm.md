# Команда /swarm — запуск роя исследователей (dual-node)

## Назначение

Запускает параллельный рой Scout-исследований по нескольким направлениям
одновременно. Реализует схему «геологической разведки»: покрытие максимального
пространства темы, а не точечный поиск по известным направлениям.

Инфраструктура: две ноды Scout (:8020 и :8021), общий ChromaDB :8000,
общий PostgreSQL :5436. Нечётные задачи → :8020, чётные → :8021.

---

## Схема параллелизма (обязательно соблюдать)

Рой выполняется в три волны — нельзя смешивать волны.

**Волна 1 — сбор URL (все задачи параллельно):**
Все N задач запускаются одновременно через N параллельных агентов.
URL-агенты работают на стороне Claude Code, не зависят от Scout-нод.
Максимум 10 параллельно — ограничение rate limits WebSearch.

**Волна 2 — индексация (обе ноды параллельно, до 4 задач одновременно):**
Распределить задачи по нодам: нечётные (SW-001, SW-003...) → :8020,
чётные (SW-002, SW-004...) → :8021. Каждая нода держит до 2 задач
одновременно. Итого до 4 задач индексируются параллельно.
ChromaDB и PostgreSQL общие — обе ноды пишут в одно хранилище.
При Playwright crash — retry на той же ноде (до 3 раз), затем на другой.

**Волна 3 — генерация брифов (до 8 параллельно):**
После завершения всей индексации — бриф-агенты партиями по 8.
scout_search и бриф работают через любую ноду (видят общий ChromaDB).
Каждый агент использует системный промпт из:
`E:\Scout\.claude\prompts\swarm_brief_system_prompt.md`

## Алгоритм выполнения

### Шаг 0 — Pre-flight check (обязательно)

Перед запуском роя — обязательная проверка:
1. `run_health_check(project="scout", no_cache=True)` — обе ноды healthy
2. Smoke test: `scout_index` (1 URL) + `scout_search` на :8020 — результат без ошибок
3. Smoke test: `scout_index` (1 URL) + `scout_search` на :8021 — результат без ошибок
4. `get_docker_stats(project="scout")` — RAM < 80%, disk > 2GB свободно
5. Проверить `RERANKER_ENABLED` в .env (должен быть `true` или отсутствовать)

Если любой пункт fail — **не запускать рой**, починить сначала.

### Шаг 1 — Проверить обе ноды
```bash
curl -s http://localhost:8020/health
curl -s http://localhost:8021/health
```
Обе должны вернуть `{"status":"ok"}`. Если одна недоступна — работать
только через доступную (батчи по 2 как fallback), зафиксировать в отчёте.

### Шаг 2 — Загрузить системный промпт
Прочитать `E:\Scout\.claude\prompts\swarm_brief_system_prompt.md`
Сохранить как SYSTEM_PROMPT для Волны 3.

### Шаг 3 — Прочитать задачи роя
```bash
ls E:\Scout\Tasks\backlog\swarm_v1\
```
Собрать список SW-*.md файлов со статусом TODO.
Разделить на два списка: нечётные → NODE_A (:8020), чётные → NODE_B (:8021).

Сообщить пользователю:
```
🚀 Рой: {N} задач | Ноды: :8020 ({K} задач) + :8021 ({M} задач)
NODE_A: SW-001, SW-003, ...
NODE_B: SW-002, SW-004, ...
```

### Шаг 4 — Волна 1: параллельный сбор URL
Запустить N параллельных агентов одновременно. Каждый агент:
- Читает параметры из своего файла задачи
- Вызывает scout_research_async с auto_collect=True, auto_collect_count=200
- Фиксирует job_id (не ждёт завершения)

Ждать пока все job_id достигнут stage="urls_collected".

Сообщить: `🔍 Волна 1 завершена: URL собраны для всех {N} задач`

### Шаг 5 — Волна 2: параллельная индексация на двух нодах

Запустить одновременно две группы:

**Группа NODE_A (:8020)** — нечётные задачи, батчи по 2:
```
Партия A1: SW-001, SW-003
Партия A2: SW-005, SW-007
Партия A3: SW-009, SW-011
...
```

**Группа NODE_B (:8021)** — чётные задачи, батчи по 2:
```
Партия B1: SW-002, SW-004
Партия B2: SW-006, SW-008
Партия B3: SW-010, SW-012
...
```

Обе группы работают параллельно независимо. Каждая группа ждёт завершения
своей партии перед запуском следующей. Статус проверять через
scout_job_status — любая нода видит все job_id через общий PostgreSQL.

Сообщать прогресс каждые 2 минуты:
```
⚙️ Индексация: A1[✓] A2[...] | B1[✓] B2[✓] B3[...]
```

### Шаг 6 — Волна 3: параллельная генерация брифов
После завершения всей индексации — запустить все N бриф-агентов параллельно.
Каждый агент вызывает scout_research_async с:
- Параметрами из файла задачи
- system_prompt = SYSTEM_PROMPT
- model = "opus"
- Нода: любая доступная (обе видят общий ChromaDB)

Ждать завершения всех брифов, сообщать прогресс каждые 3 минуты.

### Шаг 7 — Сохранить результаты
Для каждой завершённой задачи:
1. Скопировать бриф с сервера в `E:\Diser\literature\` (english или russian)
2. Обновить статус в файле задачи: TODO → DONE
3. Добавить строку в `E:\Diser\notes\research_synthesis\embedding_map_v1.md`

### Шаг 8 — Итоговый отчёт
```

Направлений: {N} | NODE_A: {K} задач | NODE_B: {M} задач
Документов: {сумма} | Чанков: {сумма}
Время индексации: ~{X} мин (было бы ~{2X} мин на одной ноде)

По задачам:
{SW-001} [:8020] — {N docs}, {N chunks} → {путь}
{SW-002} [:8021] — {N docs}, {N chunks} → {путь}
...

Результаты: E:\Diser\literature\
Карта обновлена: E:\Diser\notes\research_synthesis\embedding_map_v1.md
```

## Правила

Нода :8021 — вторая, добавлена позже. Если недоступна — деградируем
на одну ноду с батчами по 3, не останавливаем весь рой.
Нечётные → :8020, чётные → :8021 — жёсткое правило для детерминированного
распределения. Не менять без явного указания.
Максимум 2 задачи на ноду одновременно — Playwright race condition при 3+.
Системный промпт для Opus — обязателен. Без него бриф будет общим резюме.
Если задача BLOCKED — пропустить, продолжить остальные, отметить в отчёте.
