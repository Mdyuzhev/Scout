# Гибридные системы LLM + детерминированные компоненты: академические результаты

*Аналитический бриф | 2026-03-21 | Scout Research*

---

## 1. Ключевые цифры

- **pass@5 = 36.4%** — лучший результат GPT-5.2 (non-reasoning mode) в автоматической генерации формальных спецификаций для Dafny-программ на бенчмарке TESTDAFNY110. GPT-4 показал 29.1%, разница всего 7.3 п.п. несмотря на разницу поколений (arxiv, 2601.12845, 2025).

- **4 типа LLM-подходов** выделены в систематическом обзоре (roadmap) LLM в тестировании ПО: LLM-Pure Prompting, Hybrid Prompting (LLM + SOTA-инструмент), LLM-Pure Fine-tune, Hybrid Fine-tune. Pure-prompting доминирует по количеству публикаций, но гибридные подходы показывают лучшие результаты в специализированных задачах (arxiv, 2509.25043v1, 2025).

- **Снижение числа дефектов в коде** зафиксировано при интеграции GPT-3.5 Turbo / GPT-4o со статическим анализатором: итерационный цикл «статический анализ → LLM-ревизия → повторный анализ» значимо уменьшает количество багов, уязвимостей и code smells в крупномасштабных проектах (arxiv, 2506.10330v1, 2025).

- **Top-k seed selection** в фреймворке LLAMA (LLM + символическое выполнение для фаззинга смарт-контрактов) — первый подход, внедряющий обратную связь по мутациям на основе динамического feedback: адаптивная настройка вероятностей операторов мутации повышает покрытие кода (arxiv, 2507.12084, 2025).

- **Вычислительные затраты LLM** растут экспоненциально и могут достичь пика к 2026 году. По оценкам исследователей, совокупная стоимость вычислений может составить до 20% ВВП США при текущих трендах масштабирования (arxiv, 2402.00350v3, 2024).

- **Нет статистически значимого эффекта data contamination** (χ²(2)=0.79, p=0.67) при оценке LLM на задачах формальной верификации — результаты отражают реальные способности моделей, а не заучивание тренировочных данных (arxiv, 2601.12845, 2025).

- **CodeRAG** — система поиска кода на основе двудольного графа (requirement graph + DS-code graph) с хранением в Neo4j — улучшает генерацию кода LLM на уровне репозитория за счёт структурированного извлечения зависимостей и семантически близких фрагментов (arxiv, 2504.10046v1, 2025).

## 2. Основные тренды

### 2.1. Гибридные системы как доминирующая архитектура

Систематический обзор LLM в тестировании ПО (ICST/ICSE roadmap, 2025) явно выделяет «Hybrid Prompting» и «Hybrid Fine-tune» как отдельные архитектурные классы наравне с чистыми LLM-подходами. Хотя pure-prompting пока преобладает по числу публикаций, авторы отмечают, что гибридные подходы «часто интегрируются с существующими SOTA-инструментами или применяются параллельно с ними». Направление развития — не замена традиционных инструментов на LLM, а их синергия, где LLM берёт на себя генерацию и интерпретацию, а детерминированные компоненты обеспечивают верификацию и корректность.

### 2.2. Формальная верификация как feedback loop для LLM

Работа по Dafny AI Assistant демонстрирует замкнутый цикл: LLM генерирует формальные аннотации (предусловия, постусловия, инварианты циклов), верификатор Dafny проверяет их корректность, диагностика возвращается в промпт для следующей итерации. Это нейросимволический паттерн, где символический компонент (формальный верификатор) выступает арбитром качества LLM-генерации. При неудаче верификации пользователю предъявляется версия с максимальным числом доказанных обязательств — прагматичный подход вместо бинарного «прошёл/не прошёл».

### 2.3. Статический анализ как детерминированный фильтр галлюцинаций

Интеграция LLM с фреймворками статического анализа (SonarQube и аналоги) формирует архитектуру, где LLM предлагает исправления кода, а статический анализатор верифицирует их корректность. Для борьбы с галлюцинациями разработано специальное приложение «Code Comparison App», которое выявляет и корректирует ошибочные изменения до их применения к кодовой базе. RAG дополнительно повышает релевантность исправлений, обеспечивая доступ к актуальной внешней информации. Итерационный промпт-инжиниринг гарантирует соответствие выходов требованиям проекта.

### 2.4. Символическое выполнение как fallback при стагнации LLM

Фреймворк LLAMA для фаззинга смарт-контрактов демонстрирует архитектуру с явным переключением между режимами: LLM генерирует начальные seed-значения через иерархическую стратегию промптинга, но при преждевременной сходимости (path stagnation) активируется символическое выполнение для генерации новых входных данных. Это первый пример явного критерия перехода между LLM и детерминированным компонентом в контуре обратной связи — символическое выполнение включается селективно, только при исследовании глубоких путей контракта, что экономит CPU и память.

### 2.5. Граф зависимостей как структурированный контекст для LLM

CodeRAG строит двудольный граф: граф требований (requirement graph) и граф кода с зависимостями (DS-code graph), хранимый в Neo4j. LLM получает не произвольные фрагменты из RAG, а структурированные: вызываемые API, семантически близкие реализации, файлы локального контекста. Агентный reasoning-процесс позволяет LLM самостоятельно запрашивать дополнительные фрагменты через три инструмента (web search, graph reasoning, code testing). Детерминированный компонент (граф) направляет и ограничивает поиск LLM.

## 3. Гибридные архитектуры

### 3.1. LLM + статический анализ кода

**Архитектура:** Статический анализатор (SonarQube-подобный) обнаруживает дефекты → информация о каждом дефекте структурируется → LLM (GPT-3.5 Turbo / GPT-4o) генерирует исправление → «Code Comparison App» проверяет корректность изменений → повторный статический анализ подтверждает устранение дефекта.

**Ключевые элементы:** RAG для доступа к актуальной документации; итерационный промпт-инжиниринг; детерминированная валидация через Code Comparison App (сравнение diff до/после). Подход зафиксировал значимое снижение числа дефектов при повторном сканировании.

### 3.2. LLM + формальная верификация (Dafny)

**Архитектура:** LLM генерирует формальные аннотации (pre/postconditions, loop invariants) → верификатор Dafny проверяет корректность → при неудаче диагностика включается в промпт → LLM исправляет аннотации → цикл повторяется до успеха или исчерпания лимита.

**Инструмент:** Dafny AI Assistant — расширение VSCode, поддерживающее OpenAI, Anthropic, DeepSeek, xAI через единый интерфейс с приоритизацией провайдеров и автоматическим failover. GPT-5.2 достиг pass@5 = 36.4% на TESTDAFNY110.

### 3.3. LLM + символическое выполнение (Hybrid Fuzzing)

**Архитектура (LLAMA):** Три модуля: (1) LSG — LLM генерирует seed через иерархический промптинг с извлечением критических путей из смарт-контракта; (2) MOS — мульти-feedback оптимизация (seed generation, seed selection, mutation scheduling); (3) HFE — гибридный фаззинг-движок, где символическое выполнение включается селективно при стагнации путей.

**Замкнутый контур:** Результаты фаззинга возвращаются как feedback в промпт-конструкцию LLM для итеративного улучшения seed. Первый подход с адаптивной настройкой вероятностей операторов мутации на основе динамической обратной связи.

### 3.4. LLM + граф зависимостей (нейросимволическая система)

**Архитектура (CodeRAG):** Двудольный граф (requirement graph ↔ DS-code graph) хранится в Neo4j. Узлы содержат индексы, а не исходный код. Семантические связи строятся автоматически через embedding-модель (cosine similarity). LLM использует три инструмента: web search, graph reasoning, code testing — для агентного поиска поддерживающего кода.

**Принцип:** Детерминированный граф ограничивает и направляет LLM, исключая случайный поиск. LLM получает контекстуализированные фрагменты: вызываемые API, семантически похожий код, содержимое локального файла.

### 3.5. LLM + rule engines (классификация из Roadmap)

Систематический обзор LLM в тестировании выделяет «Hybrid Prompting» — комбинация необученного LLM с SOTA-инструментом (статический анализатор, символический исполнитель, генератор тестов) и «Hybrid Fine-tune» — дообученный LLM в связке с SOTA-инструментом. Обе архитектуры представлены во всех категориях тестирования: unit test generation, fuzzing, test oracles, system testing, regression testing.

## 4. Результаты сравнений

### Гибрид vs. чистый LLM

| Метрика | Чистый LLM | Гибрид | Источник |
|---------|-----------|--------|----------|
| Формальная верификация (pass@5) | 36.4% (GPT-5.2 pure) | Выше при итеративной обратной связи с Dafny verifier | arxiv 2601.12845 |
| Обнаружение дефектов кода | LLM может генерировать plausible-but-incorrect fixes (галлюцинации) | Статический анализатор + Code Comparison App отсекает ошибочные исправления | arxiv 2506.10330v1 |
| Покрытие кода при фаззинге | LLM-based fuzzers: лучше API coverage, чем традиционные | LLAMA (LLM + symbolic execution): глубже path coverage при стагнации | arxiv 2507.12084 |
| Генерация кода на уровне репозитория | Контекст ограничен окном модели | CodeRAG: структурированные зависимости из графа повышают качество | arxiv 2504.10046v1 |
| Надёжность seed generation | Одиночный промпт → low-quality/redundant seeds | Hierarchical prompting + pre-fuzzing + feedback loop → top-k filtered seeds | arxiv 2507.12084 |

**Ключевой вывод:** Гибридные системы превосходят чистый LLM не за счёт «лучшей генерации», а за счёт **детерминированной валидации** выхода. LLM генерирует варианты — формальный верификатор, статический анализатор или символический исполнитель отбирает корректные. Это принципиально снижает влияние галлюцинаций.

## 5. Российские работы

### ИСП РАН: Svace + LLM

В проиндексированных источниках (57 документов, 985 chunks) прямых упоминаний работ ИСП РАН по интеграции Svace с LLM не обнаружено. Однако архитектура, описанная в работе по Augmenting LLMs with Static Code Analysis (arxiv, 2506.10330v1), полностью применима к Svace: статический анализатор обнаруживает дефекты → LLM генерирует исправления → детерминированная проверка. ИСП РАН как разработчик Svace находится в естественной позиции для реализации такого гибрида, учитывая экспертизу в формальных методах и статическом анализе.

### Отечественный контекст

Из обзорных русскоязычных источников (Habr) упоминаются YandexGPT и GigaChat как готовые модели, но академических работ по их интеграции с детерминированными компонентами в проиндексированных данных не представлено. Российский рынок пока ориентирован на прикладное использование LLM (RAG, ассистенты разработчика), а не на гибридные архитектуры с формальной верификацией.

**Gap:** Отсутствие публичных российских бенчмарков по гибридным системам LLM + формальные методы — значимый пробел, особенно учитывая сильные позиции ИСП РАН в статическом анализе (Svace) и формальной верификации.

## 6. Gaps и возможности

### 6.1. Критерий перехода между LLM и детерминированным режимом

**Главный gap.** Из всех проанализированных работ только LLAMA (arxiv, 2507.12084) имеет явный критерий переключения: символическое выполнение включается при «premature convergence» (стагнации покрытия путей). Все остальные гибридные системы используют фиксированные пайплайны без динамического переключения. Не существует общей теоретической рамки для принятия решения: «когда LLM, когда детерминированный компонент». Это открытая исследовательская проблема.

### 6.2. Стоимостная модель гибридных систем

Ни одна из проанализированных работ не содержит TCO-анализа (total cost of ownership) гибридной системы vs. чистого LLM vs. чистого детерминированного подхода. Вычислительные затраты LLM растут экспоненциально, но конкретные бенчмарки стоимости гибридных архитектур отсутствуют.

### 6.3. Масштабируемость формальной верификации

Работа по Dafny AI Assistant явно указывает на ограничение: результаты получены на относительно небольших программах (TESTDAFNY110), и «practical problems may be more challenging». Масштабирование LLM + формальная верификация на промышленные кодовые базы — нерешённая задача. Авторы предлагают divide-and-conquer подход, но без экспериментальных данных.

### 6.4. Отсутствие стандартных бенчмарков для гибридных систем

Обзор LLM Testing Roadmap отмечает, что критерии оценки «mainly customized for different testing objectives». Единого бенчмарка, позволяющего сравнить гибридную систему с чистым LLM на стандартизированном наборе задач, не существует. Каждая работа использует собственный набор метрик.

### 6.5. Замкнутый контур обратной связи как паттерн

Все успешные гибридные архитектуры используют замкнутый контур: LLM генерирует → детерминированный компонент проверяет → результат проверки возвращается LLM. Однако оптимальное число итераций, критерии остановки и стратегии деградации (graceful degradation при исчерпании retry-лимита) исследованы минимально.

### 6.6. Возможности

- **Svace + LLM** — готовая ниша для ИСП РАН: архитектура описана, аналоги реализованы, российский статический анализатор мирового уровня.
- **Универсальный критерий переключения** — формализация метрики «когда переходить от LLM к детерминированному компоненту» (confidence score, coverage stagnation, verification failure rate).
- **Cost-aware hybrid routing** — система, которая направляет задачу в LLM или детерминированный компонент на основе прогнозируемой стоимости и ожидаемого качества.
- **Бенчмарк HybridBench** — стандартизированный набор задач для сравнения pure-LLM vs hybrid на единой метрике.
