# Метрики качества генерации кода: что и как измеряют

**Дата:** 2026-03-21
**Источники:** 57 документов, 458 chunks, 14 top-relевантных
**Тема:** SC-028 — Code Quality Metrics

## 1. Ключевые цифры

1. **164 задачи** содержит бенчмарк HumanEval (OpenAI) — де-факто стандарт для оценки генерации кода на уровне функций, хотя критикуется за простоту задач и ограничение одним языком (Python). *(pvsm.ru, 2025)*

2. **974 задачи** включает набор MBPP (Google) — краудсорсинговые задачи на Python начального уровня с 3 модульными тестами на каждую задачу. Используется как дополнение к HumanEval для более широкого покрытия. *(pvsm.ru, 2025)*

3. **1 140 задач** входит в BigCodeBench — бенчмарк нового поколения (июнь 2024), заявленный как замена HumanEval. Каждая задача содержит в среднем **5,6 тестовых случаев** и охватывает вызовы функций из **139 библиотек**. *(pvsm.ru, 2025)*

4. **1 000 задач** в DS-1000 (ноябрь 2022, Стэнфорд / Meta AI) — бенчмарк для научных данных на Python, основанный на 451 реальном вопросе со StackOverflow, покрывающий 7 библиотек. *(pvsm.ru, 2025)*

5. **133 задачи** по редактированию кода и **89 задач** по рефакторингу включает бенчмарк Aider — open-source решение для оценки LLM как парного программиста. Модели получают одну попытку на исправление, если тесты не пройдены с первого раза. *(pvsm.ru, 2025)*

6. **Простые изменения в тестах** бенчмарка (например, перестановка порядка вопросов) могут сдвинуть рейтинг модели на **до 8 позиций**, что ставит под сомнение надёжность лидербордов. *(habr.com, 2024)*

7. **50-100 тестовых кейсов** рекомендуется как стартовый объём для оценки LLM-системы в продакшене — достаточно для выявления слабых мест, но управляемо по трудозатратам. *(habr.com/llmarena.team, 2025)*

## 2. Основные тренды

### 2.1. Переход от синтетических задач к реальным сценариям

Ранние бенчмарки (HumanEval, MBPP) оценивали способность модели написать элементарную функцию — понимает ли она синтаксис и базовые структуры языка. Современные бенчмарки (BigCodeBench, DS-1000, SWE-bench) ставят задачи на уровне реальных проектов: рефакторинг, интеграция с библиотеками, работа с существующей кодовой базой. Это отражает смену парадигмы оценки — от «может ли модель код» к «может ли модель быть полезным инженером». Бенчмарк Aider, например, оценивает именно парное программирование в терминале.

### 2.2. ИИ оценивает ИИ (LLM-as-Judge)

Растёт использование моделей-судей для автоматической оценки генерации. MT-Bench использует GPT-4 как арбитра между двумя ответами; GPTScore / G-Eval позволяют LLM оценивать код по заданным критериям (корректность, стиль, логичность). Этот подход масштабируем и позволяет обрабатывать огромные объёмы данных без утомления. Однако ИИ-оценщики сами по себе могут быть предвзятыми — например, отдавать предпочтение более длинным или более формальным ответам, что создаёт систематическое смещение в оценках.

### 2.3. Elo-рейтинги и арены для живого сравнения

Платформы вроде LMSYS Chatbot Arena адаптируют шахматную систему Elo для ранжирования LLM. Модели соревнуются на одинаковых задачах, пользователи голосуют за лучший ответ, рейтинги корректируются динамически. Этот подход особенно эффективен для выявления тонких различий между top-моделями, которые статические бенчмарки не улавливают. Arena Hard версия специально нацелена на разделение лидеров рейтинга через усложнённые задачи.

### 2.4. Проблема утечки данных (data contamination)

Ключевая уязвимость современных бенчмарков — попадание тестовых данных в корпус предобучения модели. Если модель уже «видела» задачи HumanEval при обучении, её результаты искусственно завышены. DS-1000 пытается решить эту проблему, модифицируя оригинальные задачи со StackOverflow, чтобы исключить запоминание. Для надёжной оценки рекомендуется использовать частные или свежесозданные наборы данных, хранящиеся отдельно от процесса обучения.

### 2.5. Кастомные бенчмарки для специализированных доменов

Стандартные бенчмарки не покрывают специфику конкретных технологических стеков. Показателен пример из экосистемы 1С: создан первый автоматический бенчмарк для оценки LLM в 1С-разработке с использованием метрик BLEU, CodeBLEU и pass@k, адаптированных для специфического синтаксиса платформы. Это отражает тренд на создание domain-specific evaluation, где общие бенчмарки дают слишком поверхностную картину.

## 3. Стандартные метрики генерации кода

### pass@k

Основная метрика функциональной корректности. Модель генерирует k вариантов решения для каждой задачи; pass@k — вероятность того, что хотя бы один из k вариантов пройдёт все модульные тесты. pass@1 отражает качество с первой попытки, pass@10 и pass@100 показывают потенциал модели при множественной генерации. Широко используется в HumanEval, MBPP и их производных. Адаптируется для нестандартных языков (например, 1С) путём замены тестовой инфраструктуры.

### Compilation Rate (доля компилируемого кода)

Процент сгенерированных решений, которые компилируются без ошибок. Базовая метрика, отсекающая синтаксически невалидный код до проверки логики. Для интерпретируемых языков (Python) — аналогом служит отсутствие SyntaxError при импорте/выполнении.

### Test Pass Rate

Доля модульных тестов, пройденных сгенерированным кодом. В отличие от pass@k (бинарная метрика: все тесты пройдены или нет), test pass rate может быть дробным — например, 7 из 10 тестов. Позволяет оценивать частичную корректность. В бенчмарке BigCodeBench среднее число тестов на задачу — 5,6, что делает эту метрику более гранулярной.

### Acceptance Rate

Метрика из IDE-интеграций (Copilot, Cline и др.): доля предложений модели, которые разработчик принял без изменений. Отражает практическую полезность в реальном workflow, но субъективна — зависит от опыта и привычек конкретного разработчика. Не используется в академических бенчмарках, но критична для продуктовой оценки.

### BLEU / CodeBLEU

BLEU измеряет перекрытие n-грамм между сгенерированным и эталонным кодом (от 0 до 1). CodeBLEU расширяет BLEU, добавляя учёт синтаксических деревьев (AST match) и потока данных (data flow match). Используется когда нет возможности запустить код (нет тестовой среды для языка). Ограничение: не отражает функциональную корректность — код может быть семантически идентичен, но структурно отличаться.

### BERTScore

Семантическое сравнение на уровне контекстных эмбеддингов BERT. Применяется реже для кода, чем для естественного языка, но полезен для оценки комментариев, документации и описаний к функциям.

## 4. Бенчмарки

### HumanEval (OpenAI, 2021)

Набор из 164 задач на Python. Каждая — docstring + сигнатура функции, модель дописывает тело. Проверка через модульные тесты. Метрика: pass@k. Критика: задачи слишком просты, один язык, высокий риск contamination. Несмотря на это, остаётся самым цитируемым бенчмарком для code generation.

### MBPP (Google, 2021)

974 задачи Python начального уровня с 3 тестами на задачу. Краудсорсинговый, покрывает стандартные библиотеки. Дополняет HumanEval по объёму, но также критикуется за простоту.

### BigCodeBench (2024)

1 140 задач, в среднем 5,6 тестов, 139 библиотек. Позиционируется как «новый HumanEval» с более сложными инструкциями и множественными вызовами функций. Предоставляет примеры от open-source LLM.

### MultiPL-E (2022)

~530 задач, расширение HumanEval на множество языков программирования. Позволяет сравнивать модели кросс-лингвистически.

### DS-1000 (Стэнфорд / Meta AI, 2022)

1 000 задач Python для научных данных (NumPy, Pandas, sklearn и др.). Основан на реальных вопросах StackOverflow с модификациями против запоминания.

### Aider Benchmark (open-source)

133 задачи на редактирование + 89 на рефакторинг кода Python. Уникальность: оценивает модель как парного программиста в терминале с возможностью одного исправления. Имеет собственный лидерборд.

### SWE-bench

Бенчмарк для оценки способности моделей решать реальные GitHub issues. Задачи взяты из настоящих pull requests open-source проектов. Считается наиболее приближённым к реальной работе разработчика.

### LiveCodeBench

Динамический бенчмарк, регулярно обновляемый новыми задачами для борьбы с contamination. Задачи берутся из свежих соревнований по программированию, которых не было в обучающих данных моделей.

## 5. Стабильность и variance

Проблема повторяемости результатов — одна из наименее исследованных, но наиболее практически значимых.

**Факт о нестабильности рейтингов:** простые изменения в бенчмарке (перестановка порядка вопросов, изменение формулировок) сдвигают позиции моделей на до 8 мест. Это означает, что заявленный рейтинг модели может не отражать её реальное положение.

**Непостоянная производительность LLM:** источники фиксируют, что модели «в одну минуту пишут как профессионалы, в следующую — совершают глупые ошибки». Эта стохастичность заложена в архитектуру (температура сэмплирования, top-p), но бенчмарки обычно фиксируют один набор гиперпараметров, не отражая полный спектр variance.

**pass@k как частичное решение:** метрика pass@k при k > 1 косвенно учитывает variance — если pass@1 = 0.5, а pass@10 = 0.95, это говорит о высокой дисперсии качества генерации. Однако в продакшене пользователь обычно видит один ответ (k=1), что делает variance критичной.

**Методологическая проблема:** большинство бенчмарков публикуют результаты одного прогона без confidence intervals. Нет общепринятого стандарта для указания стандартного отклонения или числа повторений. Это затрудняет статистически корректное сравнение моделей.

## 6. Gaps и возможности

1. **Отсутствие метрик maintainability.** Существующие бенчмарки проверяют, работает ли код, но не оценивают читаемость, архитектурную чистоту, следование паттернам проектирования. Код, прошедший все тесты, может быть unmaintainable.

2. **Слабое покрытие языков и экосистем.** Подавляющее большинство бенчмарков — Python-only. Языки вроде Go, Rust, Kotlin, Swift, а тем более нишевые платформы (1С, ABAP, PL/SQL) практически не представлены в стандартных наборах.

3. **Нет стандартов для оценки variance.** Не существует общепринятого протокола, требующего N повторных прогонов и публикации confidence intervals. Результаты одного запуска публикуются как финальный рейтинг.

4. **Security и safety метрики для кода.** Сгенерированный код может содержать SQL-инъекции, race conditions, утечки памяти. Систематической проверки безопасности сгенерированного кода в существующих бенчмарках нет.

5. **Оценка в контексте проекта.** Все бенчмарки работают на изолированных задачах. Способность модели понять существующую кодовую базу в 100K+ строк и внести согласованное изменение — практически не измеряется (SWE-bench — первый шаг в этом направлении).

6. **Метрики для не-Python языков в data science.** DS-1000 покрывает Python, но R, Julia, SQL для аналитических задач — белое пятно.

7. **Долгосрочная корректность.** Бенчмарки тестируют код в момент генерации. Как сгенерированный код ведёт себя при обновлении зависимостей, под нагрузкой, в edge cases — не отслеживается.
