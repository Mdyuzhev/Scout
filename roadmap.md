# Scout — Roadmap

> Живой документ. Вектор важнее деталей — детали уточняются перед каждой волной.

---

## Идея

Scout — MCP-сервер предобработки данных для продуктовых исследований.
Агент настраивает инструмент под задачу через `ResearchConfig`, Scout собирает
и фильтрует данные детерминированно, агент получает готовую выжимку и синтезирует.

**Принцип**: LLM — последний шаг, не первый.

---

## Ключевые архитектурные решения (зафиксированы)

**Язык**: Python 3.12 — экосистема ML, переиспользование RAG-QA компонентов.

**Хранилище истории**: PostgreSQL (порт 5436) — полнотекстовый поиск по темам
через `tsvector` + `GIN`-индексы, `JSONB` для гибкого хранения `ResearchConfig`.
"Сначала проверь историю" — перед любым новым сбором данных Scout ищет похожую
завершённую сессию в БД.

**Векторный индекс**: ChromaDB (persistent) — эмбеддинги чанков для семантического
поиска. Каждая сессия = отдельная коллекция `session_{id}`.

**Разделение**: Postgres хранит метаданные и историю, ChromaDB — векторы.
Вместе они образуют data lake истории исследований.

**Расширяемость**: коллекторы, чанкеры и briefer'ы — стратегии (BaseCollector,
BaseChunker, BaseBriefer). Новый источник = новый файл, без правок в ядре.

**MCP-интерфейс**: три укрупнённых инструмента (`scout_index`, `scout_search`,
`scout_brief`) + вспомогательные (`scout_status`, `scout_list_sessions`).

## Wave 1 — POC (текущая)

> Цель: end-to-end пайплайн на реальных данных. Не production, но честно работающий.
> Ключевой вопрос POC: убедиться что связка web→chunk→embed→search→brief даёт
> полезный результат на реальной исследовательской задаче.

- [ ] **SC-001** — Scaffold: структура проекта, `requirements.txt`, `docker-compose.yml`, `.env.example`
- [ ] **SC-002** — Модели данных: `ResearchConfig`, `ResearchSession`, `ResearchPackage` (Pydantic)
- [ ] **SC-003** — Ingestion: `WebCollector` (httpx + BS4), `SlidingWindowChunker`, `Indexer` (ChromaDB)
- [ ] **SC-004** — Retrieval: `Searcher` (порт из RAG-QA), `ContextBuilder` → `ResearchPackage`
- [ ] **SC-005** — MCP-сервер: `scout_index` / `scout_search` / `scout_brief` (FastMCP 2.0)
- [ ] **SC-006** — PostgreSQL: схема сессий, `SessionStore`, логика "сначала проверь историю"
- [ ] **SC-007** — Deploy: Dockerfile, CI/CD (GitHub Actions), `/opt/scout` на сервере

**Результат Wave 1**: агент вызывает `scout_index(config)` → Scout ищет историю →
если нет — собирает, индексирует → агент вызывает `scout_search(query)` →
получает `ResearchPackage` → пишет отчёт через `scout_brief`.

## Wave 2 — Quality & Reliability

> Цель: улучшить качество поиска, сделать систему надёжнее.

- Двухуровневая фильтрация: широкий захват (top-50) → reranker → top-15
- Дедупликация источников по хэшу содержимого (не только по URL)
- Оценка качества источника: авторитетность домена, свежесть, тип контента
- Поддержка локальных документов (txt, md, pdf) как источников — `LocalFileCollector`
- Настраиваемый TTL кэша: не пересобирать индекс если сессия свежая

---

## Wave 3 — Integration & UX

> Цель: Scout как инструмент первого класса в рабочем процессе.

- `scout_brief` с выбором LLM: Anthropic Haiku (качество) / Ollama mistral (offline)
- Интеграция в LocOll: виджет истории исследований из браузера
- `scout_list_sessions` с фильтрацией по теме, дате, статусу
- API для просмотра и переиспользования старых сессий
