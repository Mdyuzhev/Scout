# Scout — Roadmap

> Живой документ. Вектор важнее деталей — детали уточняются перед каждой волной.

---

## Идея

Scout — MCP-сервер предобработки данных для продуктовых исследований.
Агент настраивает инструмент под задачу через `ResearchConfig`, Scout собирает
и фильтрует данные детерминированно, агент получает готовую выжимку и синтезирует.

**Принцип**: LLM — последний шаг, не первый.

---

## Ключевые архитектурные решения (зафиксированы)

**Язык**: Python 3.12 — экосистема ML, переиспользование RAG-QA компонентов.

**Хранилище истории**: PostgreSQL (порт 5436) — полнотекстовый поиск по темам
через `tsvector` + `GIN`-индексы, `JSONB` для гибкого хранения `ResearchConfig`.
"Сначала проверь историю" — перед любым новым сбором данных Scout ищет похожую
завершённую сессию в БД.

**Векторный индекс**: ChromaDB (persistent) — эмбеддинги чанков для семантического
поиска. Каждая сессия = отдельная коллекция `session_{id}`.

**Разделение**: Postgres хранит метаданные и историю, ChromaDB — векторы.
Вместе они образуют data lake истории исследований.

**Расширяемость**: коллекторы, чанкеры и briefer'ы — стратегии (BaseCollector,
BaseChunker, BaseBriefer). Новый источник = новый файл, без правок в ядре.

**MCP-интерфейс**: три укрупнённых инструмента (`scout_index`, `scout_search`,
`scout_brief`) + вспомогательные (`scout_status`, `scout_list_sessions`).

**Фронтенд**: Scout Web UI — отдельный сервис, не интегрируется в LocOll.
Отдельный контейнер в том же `docker-compose.yml`, отдельный порт (8030).
Фронт общается с scout-mcp через REST API (`/api/*` custom_route на том же порту 8020).

## Wave 1 — POC ✅ ЗАВЕРШЕНА

Все SC-001..SC-015 выполнены. Пайплайн работает end-to-end на реальных данных —
русскоязычных и англоязычных источниках, двух разных темах, с брифом без галлюцинаций.

**Итоговые метрики Wave 1:**

| Прогон | Docs | Chunks | Brief tokens | Язык |
|--------|------|--------|-------------|------|
| SC-012 analytics (190 URL) | 109 | 562 | — | EN |
| SC-013 analytics (363 URL) | 204 | 1031 | — | EN |
| SC-014 AI tools (314 URL) | 184 | 746 | 9 436 | EN |
| SC-015 Ростелеком (136 URL) | 65 | 299 | 21 548 | RU |

**Ключевые открытия Wave 1:**
- DDG HTML scraping нежизнеспособен на сервере — URL-режим стал основным флоу
- Diminishing returns по качеству поиска наступает после ~150-200 документов
- `paraphrase-multilingual-MiniLM-L12-v2` одинаково хорошо работает на RU и EN
- FastMCP 2.0 требует `initialize` → `Mcp-Session-Id` перед tool calls
- SSE-ответ парсится построчно, результат в `result["result"]["structuredContent"]`
- Перезапуск контейнера через `--force-recreate --no-deps`, не `docker restart`

## Wave 2 — Quality & Reliability

> Цель: улучшить качество поиска и надёжность пайплайна. Только бэкенд, без UI.
> Главная задача: поднять качество брифа через лучший отбор контекста.

**Реранкер** — самое важное улучшение. Текущая cosine similarity даёт плоский
диапазон 0.65–0.80, где разница между релевантным и нерелевантным чанком ~0.1.
Cross-encoder модель (`cross-encoder/ms-marco-MiniLM-L-6-v2`) оценивает пару
запрос+документ напрямую — это точнее. Схема: широкий захват top-50 через ChromaDB,
затем реранкер отбирает top-15 для брифа.

**Оценка качества источника** — фильтрация мусора до индексации. Признаки плохого
источника: текст < 300 слов после очистки, доля навигационных элементов высокая,
домен из стоп-листа (G2, Capterra, агрегаторы с bot-protection). Хорошего:
плотность ключевых сущностей, наличие чисел и конкретики, авторитетный домен.

**Дедупликация по содержимому** — сейчас дедупликация по URL. Нужна по хэшу
содержимого ещё на этапе фетча, чтобы не индексировать перепечатки и зеркала.

**LocalFileCollector** — поддержка локальных документов (txt, md, pdf) как источников.
Нужна для исследований по внутренним документам и отчётам.

**Настраиваемый TTL** — `cache_ttl_hours: 0` сейчас форсирует пересбор. Нужна
умная инвалидация: если тема "горячая" (новости, продукт), TTL короткий; если
"вечная" (технология, принципы), TTL долгий или ручная инвалидация.

## Wave 3a — REST API (пре-условие для фронта)

> Цель: добавить HTTP REST API поверх MCP-сервера для нужд фронтенда.
> Фронт не умеет в MCP-протокол — ему нужен обычный JSON API.

Реализуется через `@mcp.custom_route` в `mcp_server.py` — без отдельного сервиса.
Все эндпоинты на том же порту 8020.

Минимальный набор для фронта:

`GET /api/sessions` — список сессий (с пагинацией, фильтрацией по теме/статусу).
`GET /api/sessions/{id}` — детали сессии + бриф если есть.
`POST /api/index` — запустить индексацию (async, возвращает session_id сразу).
`GET /api/sessions/{id}/status` — polling статуса долгой операции.
`POST /api/search` — поиск по сессии.
`POST /api/brief` — сгенерировать бриф.
`DELETE /api/sessions/{id}` — удалить сессию + ChromaDB коллекцию.

Аутентификация — простой Bearer токен из `.env` на первом этапе.

## Wave 3b — Scout Web UI

> Цель: браузерный интерфейс для запуска и просмотра исследований.
> Отдельный сервис, отдельный контейнер, не связан с LocOll.

**Стек — выбор между двумя подходами:**

Go + htmx (по паттерну LocOll) — если UI простой: запустить исследование,
посмотреть прогресс, прочитать бриф, экспортировать. Лёгкий, без npm, быстро.

React / Next.js — если UI аналитический: история сессий с поиском и фильтрами,
сравнение брифов, управление URL-списками, drag-and-drop. Сложнее, но мощнее.

Решение принять в начале Wave 3b — зависит от того какой сценарий использования
окажется главным после Wave 2.

**Ключевые экраны:**
- Главная: история исследований, поиск по теме
- Новое исследование: поле темы, URL-список (textarea или upload), параметры
- Сессия: прогресс индексации, результаты поиска, бриф, экспорт в MD/PDF
- Настройки: API ключ, дефолтные параметры, модель

**Деплой**: отдельный контейнер `scout-web` в `docker-compose.yml`, порт 8030.
CI/CD через тот же GitHub Actions workflow — собирает оба контейнера.

## Wave 4 — Scout как продукт

> Цель: Scout работает самостоятельно, не только по запросу.

**Мониторинг тем** — периодический пересбор "подписанных" тем. Агент получает
уведомление когда появились новые значимые материалы. Хранится в PostgreSQL.

**Специализированные коллекторы** — Habr (важен для RU технического контента),
YouTube transcript (через API), Reddit, Telegram-каналы. Каждый = отдельный
BaseCollector без правок в ядре.

**Экспорт** — бриф в Markdown, PDF (через weasyprint), DOCX. Нужен для передачи
результатов стейкхолдерам которые не работают в IDE.

**Сравнение сессий** — взять две сессии по похожим темам и попросить Haiku/Sonnet
выделить различия. Полезно для отслеживания изменений рынка со временем.

**Выбор модели для брифа** — сейчас только Haiku. Нужна возможность выбрать
Sonnet для сложных тем где нужна глубина, или Ollama mistral для offline/приватного.
